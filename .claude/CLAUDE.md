# エージェントへの指示 (Agent Instructions)

このファイルは、FamNoteプロジェクトにおけるAIエージェントの振る舞いとコーディングスタイルを定義するものです。

## 1. コミュニケーションと言語 (Communication & Language)
- **完全な日本語対応:** ユーザーへの返答、コード内のコメント、コミットメッセージ、内部タスクリスト等すべてのテキストは**必ず日本語**で行ってください。
- **プロフェッショナルな対応:** フレンドリーかつ専門的なトーンを保ち、技術的な提案を積極的に行ってください。

## 2. デザインと UI/UX (Design and UI/UX)
- **プレミアムなデザイン:** 洗練されたモダンなUIを心がけてください。
  - 配色: ダークモード対応、Tailwindのslate/zinc基調 + グリーン/アンバーのアクセントカラー
  - フォント: Inter（見出し）+ Noto Sans JP（日本語）
  - アニメーション: Framer Motion で画面遷移・カード出現・数値カウントアップ
  - マイクロインタラクション: ボタンホバー・タップフィードバック・スケルトンローディング
- **レスポンシブデザイン:** モバイルファーストで実装。

### テーマシステム（SkillSync準拠）
- **CSS変数**: `--color-brand-primary` / `--color-brand-secondary`
- **デフォルト**: オレンジ `#E85513` × ネイビー `#00133F`
- **ユーザー選択**: J1リーグ20チームのカラーパレット（有料プランのみ）
- **保存先**: localStorage `famnote_theme`
- **環境変数プレフィックス**: 公開可能な値のみ `VITE_` を付ける（秘密鍵は付けない）

## 4. コーディングガイドライン

- `any` 型の使用は禁止。適切な型定義を行うこと。
- Server Components / Client Components を適切に使い分ける（`"use client"` は必要な箇所のみ）。
- コンポーネントは単一責任原則に従い小さく分割。UIとロジック（Hooks）を分離。
- エラー発生時はトースト通知でユーザーに表示。
- ライブラリインストール前にセキュリティ脆弱性を確認する。
- 機能実装・修正後は必ず単体テストを実装・修正する。
- XSS・インジェクション・認証バイパス対策を徹底する。
- Stripe Webhookは必ず署名検証を行う。

## 5. プロセスと進捗管理

- **自律実行:** 実装・テスト・デプロイは承認なしで自律的に進める。ユーザー確認が必要なのは以下のみ：
  - APIキー・シークレット・認証情報の変更
  - Firestoreルールの権限大幅緩和
  - Stripe課金ロジックの変更
- **アーティファクトの更新:** `task.md` / `implementation_plan.md` を常に最新に保つ。
- **デグレード防止:** 既存機能への影響がないことを確認してから実装する。

## 6. サブエージェント (Sub-Agents)

新機能開発には以下の自己改善ループを使用してください。

### エージェント構成

| エージェント | 役割 |
|------------|------|
| **planner** | 機能要求 → 詳細仕様書の作成 |
| **designer** | 仕様書 → UI/UXデザイン仕様・デザイントークン・コンポーネント設計 |
| **generator** | 仕様書＋デザイン仕様 → 実装・ユニットテスト・E2Eテスト |
| **evaluator** | 実装の多角的評価・フィードバック・自己改善ループ制御 |

### 自己改善ループ

```
ユーザーの機能要求
    ↓
[Planner] 仕様書作成（承認不要・自動続行）
    ↓ ─── 🛑 STOP/PAUSE で停止可能
[Designer] UI/UXデザイン仕様作成
    ↓ ─── 🛑 STOP/PAUSE で停止可能
[Generator] 実装・ユニットテスト・E2Eテスト作成
    ↓ ─── 🛑 STOP/PAUSE で停止可能
[Evaluator] ビルド・テスト・コードレビュー・セキュリティ・UX評価
    ↓ 不合格（ループ回数 < 3）
[Generator] フィードバックをもとに修正 → [Evaluator] 再評価
    ↓ 合格 または ループ3回到達
/commit（合格時）または ユーザーへエスカレーション（上限到達時）
```

### 🛑 人間による停止トリガー

| コマンド | 動作 |
|---------|------|
| `STOP` | 現在のループを即座に停止し、状況をレポートする |
| `PAUSE` | 次のエージェント起動前に確認を求める |
| `ROLLBACK` | 最後のgit commitを取り消してループを停止する |

### 自己改善ループの自律制御ルール

- Evaluatorが合格を出すまでGenerator→Evaluatorを最大**3回**繰り返す
- 3回で合格しない場合は自動停止してユーザーにエスカレーション報告する
- 各エージェント起動前に `.claude/loop_state.json` の `stop_requested` フラグを確認する
- ループ状態は `.claude/loop_state.json` に記録して進捗を可視化する

### ループ状態ファイル形式（`.claude/loop_state.json`）

```json
{
  "feature": "機能名",
  "phase": "planner|designer|generator|evaluator",
  "loop_count": 0,
  "max_loops": 3,
  "stop_requested": false,
  "history": []
}
```

### 使用タイミング
- 新機能追加・大きな改修（4ファイル以上）→ フルループを使用
- 小さな修正・バグ修正 → 従来のスキルフローを使用

## 7. スキルの使用ルール (Skills)

| コマンド | 使うタイミング | 省略できる条件 |
|---------|--------------|--------------|
| `/gather-requirements` | 要求が曖昧・複数の解釈がある場合 | 要件が明確な修正はスキップ可 |
| `/create-plan` | 新機能追加・4ファイル以上の大きな改修の前 | 3ファイル以内の小修正はスキップ可 |
| `/run-tests` | 実装完了後・コミット前（**必須**） | 省略不可 |
| `/security-check` | 認証・Stripe・外部API・ユーザー入力を扱う変更後 | UIのみ・スタイル修正はスキップ可 |
| `/sync-docs` | 新機能追加・API仕様変更・画面構成の大幅変更後 | バグ修正・UI調整はスキップ可 |
| `/commit` | 実装完了後（**必須**） | 省略不可 |

## 8. ドキュメント生成
